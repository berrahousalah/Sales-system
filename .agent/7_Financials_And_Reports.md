# Financials & Reports Page Logic (Financials_And_Reports.md)

## 1. General Concept
* Professional enterprise architectures do not rely on completely fluid, real-time recalculations for historic data. They utilize a distinct separation between dynamic discovery views and fixed, unalterable accounting snapshots.
* This module prevents "data drift" (where modifying a past transaction retrospectively changes the historical revenue of a prior month) by introducing a strict **Financial Closing (Month-End Close)** mechanism and a rolling **Transaction Ledger**.

## 2. Dynamic Real-Time Reports Screen
This interface is utilized for open-ended, fluid operations and daily analytical discovery.
* **UI Controls:** A date range filter input (From `Date X` to `Date Y`).
* **Real-Time Computations:** The data updates organically based on live transaction values, reflecting any returns or invoice adjustments executed on that day.
* **Financial Metrics Specifications:**
  * **Gross Revenue:** Cumulative sum of all retail sales generated within the designated window (aggregating both instant Cash sales and Accounts Receivable debts).
  * **Cost of Goods Sold (COGS):** Total actual historical procurement cost extracted directly from the specific product batches associated with the items sold during the filtered timeframe.
  * **Operating Expenses (OPEX):** Consolidated sum of shipping/loading costs incurred via Import Invoices, plus the absolute dollar loss value generated by stock write-offs within the `Inventory Adjustments` module.
  * **Gross and Net Profit Calculations:** 
    * `Gross Profit = Gross Revenue - COGS`
    * `Net Profit = Gross Profit - Operating Expenses`

## 3. Financial Closing & Hard Archiving Engine
To ensure past financial reporting integrity, the system implements a rigid closing framework that stops past numbers from drifting due to historical updates.

* **UI Trigger:** A prominent "Month-End Close" button, typically executed on the final calendar day of a monthly cycle.
* **Backend Processing Sequence:**
  1. **State Snapshot Generation:** The system locks down and calculates the final numbers for that specific month.
  2. **Immutable Record Writing:** The backend writes these metrics to a dedicated, read-only database table (`MonthlyFinancialReports`).
  3. **Database Schema Constraints:** Records written to this table must be fundamentally **Immutable (Non-updateable / Non-deletable)**. The table maps fields explicitly for: `ClosingMonthId`, `TotalRevenue`, `TotalCOGS`, `TotalLogisticsExpenses`, `TotalAdjustmentLosses`, and `FinalNetProfit`.
* **The Financial Archive View:**
  * Displays a historical timeline log of all closed months/years.
  * Clicking an archive entry (e.g., "June 2026") pulls the static snapshot directly from the immutable database record, rendering a static document view with an export-to-PDF payload. This report remains identical forever, regardless of any later data manipulation in the active system.

## 4. Retroactive Adjustments & Transaction Ledger Framework
When a merchant modifies a historical document (e.g., reducing an item quantity or executing a return on a Sales Invoice created in July, while the current active month is August and July has already been closed):
* **The Core Rule:** The closed historical snapshot for July is **never** altered.
* **Backend Mechanism (Transaction Ledger):**
  * The database handles financial activity via an append-only `TransactionLedger` table.
  * Instead of applying an inline edit directly to a historical total, the backend leaves the archived monthly data intact and instead logs a **Negative Financial Delta Event** timestamped with the *actual current execution date* (August).
* **The Workflow Effect:**
  * Because the negative delta entry is timestamped with the current live date, it falls naturally into the active month's dynamic reporting pool.
  * The current month's dynamic net profit organically absorbs and balances out the old loss or reduction. This protects the absolute integrity of historic tax or accounting reports while maintaining correct rolling cross-year metrics.
